rover_sim.c Function Documentation
=================================

Overview
--------
This document explains the functions in rover_sim.c, what each one does,
how it works, and what design decisions or edge cases are important.

The program simulates a Mars rover on a 50x50 grid. It loads a map from
mars_map_50x50.csv, searches for minerals, plans safe trips, manages
battery and time, and writes logs to rover_log.csv and ai_route.txt.


1. inb
------
Signature:
    static inline int inb(int x, int y)

Purpose:
    Checks whether a coordinate is inside the map bounds.

Parameters:
    x - x coordinate
    y - y coordinate

Returns:
    Nonzero if the position is inside the 50x50 map, otherwise 0.

How it works:
    It tests:
        x >= 0 && x < W
        y >= 0 && y < H

Notes:
    Used throughout pathfinding and BFS to reject out-of-bounds cells.


2. is_day_tick
--------------
Signature:
    static inline bool is_day_tick(int tick)

Purpose:
    Determines whether a simulation tick happens during daytime.

Parameters:
    tick - current half-hour simulation round

Returns:
    true if the tick is in the daytime segment of the cycle, false otherwise.

How it works:
    The day/night cycle repeats every CYCLE_TICKS.
    A tick is considered daytime if:
        (tick % CYCLE_TICKS) < DAY_TICKS

Notes:
    Tick 0 starts at sunrise.


3. exact_time_hours_sunrise
---------------------------
Signature:
    static inline float exact_time_hours_sunrise(int tick)

Purpose:
    Converts a simulation tick into a wall-clock time.

Parameters:
    tick - current half-hour simulation round

Returns:
    Time of day in hours, starting from 06:00 at tick 0.

How it works:
    It starts from 6.0 and adds 0.5 hours for each tick.
    Then it wraps the value into the [0, 24) range.

Example:
    tick 0 -> 6.0
    tick 1 -> 6.5
    tick 2 -> 7.0

Notes:
    Used for logging only.


4. clamp
--------
Signature:
    static inline double clamp(double v, double lo, double hi)

Purpose:
    Restricts a value to a specified range.

Parameters:
    v  - input value
    lo - lower bound
    hi - upper bound

Returns:
    lo if v < lo, hi if v > hi, otherwise v.

Notes:
    Mainly used to keep battery values between 0 and BAT_MAX.


5. move_cost
------------
Signature:
    static inline double move_cost(int v)

Purpose:
    Computes battery cost for moving at speed v.

Parameters:
    v - number of cells moved in one tick

Returns:
    Battery cost equal to 2 * v^2.

Examples:
    v = 1 -> 2
    v = 2 -> 8
    v = 3 -> 18

Notes:
    Higher speeds are much more expensive than lower speeds.


6. idle_cost
------------
Signature:
    static inline double idle_cost(bool mining)

Purpose:
    Returns the battery cost of a non-movement action.

Parameters:
    mining - whether the action is mining

Returns:
    2.0 if mining is true, otherwise 1.0.

Notes:
    Standing costs 1, mining costs 2.


7. solar_charge
---------------
Signature:
    static inline double solar_charge(bool day)

Purpose:
    Returns how much battery is recharged during the current tick.

Parameters:
    day - whether the current tick is daytime

Returns:
    10.0 during daytime, 0.0 at night.


8. chebyshev
------------
Signature:
    static double chebyshev(int x0, int y0, int x1, int y1)

Purpose:
    Computes Chebyshev distance between two cells.

Parameters:
    x0, y0 - start coordinates
    x1, y1 - target coordinates

Returns:
    max(abs(x1 - x0), abs(y1 - y0))

Why this is used:
    Movement is allowed in 8 directions with equal step cost, so Chebyshev
    distance is an appropriate heuristic for A*.


9. min_ticks_to_cover_steps
---------------------------
Signature:
    static int min_ticks_to_cover_steps(int steps, int v)

Purpose:
    Computes the minimum number of ticks needed to travel a given number
    of steps at speed v.

Parameters:
    steps - number of cells to travel
    v     - speed in cells per tick

Returns:
    The ceiling of steps / v.

How it works:
    Uses integer ceiling division:
        (steps + v - 1) / v


10. trip_min_ticks
------------------
Signature:
    static int trip_min_ticks(int out_steps, bool mine_pending, int in_steps)

Purpose:
    Estimates the minimum number of ticks needed to complete a trip plan.

Parameters:
    out_steps     - outward travel distance
    mine_pending  - whether mining still needs to happen
    in_steps      - return travel distance

Returns:
    Minimum ticks required assuming movement speed can be 3.

How it works:
    Adds:
        ticks to travel outward at speed 3
        + 1 tick for mining if needed
        + ticks to travel inward at speed 3

Notes:
    Used inside feasibility checks to make sure enough mission time remains.

    static void log_gui_line(FILE *f, int round, const Rover *r,

                             const char *timePeriod, float exactTime,

Purpose:
    Writes one formatted CSV log line describing the rover state.

Parameters:
    f          - output file

    r          - pointer to rover state
    speed      - speed used this tick
    pathCount  - total moved cells
    timePeriod - "DAY" or "NIGHT"

Behavior:

Notes:
    Used for both rover_log.csv and ai_route.txt.


12. precompute_home_dist
------------------------
Signature:
    static void precompute_home_dist(int8_t sx, int8_t sy)

Purpose:
    Precomputes shortest distance from every reachable cell back to home.

Parameters:
    sx, sy - home/start coordinates

Returns:
    Nothing. Fills the global home_dist array.

How it works:
    Uses breadth-first search from the home cell.
    Each valid non-obstacle neighbor gets distance d + 1.

Important details:
    - Movement is allowed in 8 directions.
    - Obstacles are skipped.
    - Unreachable cells remain 255.

Why it matters:
    It gives a fast exact step count back home without re-running pathfinding
    for every decision.

Edge cases:
    Minerals in unreachable areas are later ignored because their home_dist
    value stays 255.


13. reconstruct
---------------
Signature:
    static int reconstruct(Node nodes[H][W], int sx, int sy, int gx, int gy,
                           Pt *path, int maxlen)

Purpose:
    Rebuilds the final A* path from goal back to start using parent pointers.

Parameters:
    nodes  - A* search data
    sx, sy - start coordinates
    gx, gy - goal coordinates
    path   - output path buffer
    maxlen - maximum path length allowed

Returns:
    Path length on success, -1 on failure.

How it works:
    1. Start at the goal.
    2. Follow parent pointers backward until reaching the start.
    3. Store the reversed sequence.
    4. Reverse the array so the path is in start-to-goal order.

Output format:
    path[0] is the start cell and the last element is the goal cell.

Failure cases:
    - parent chain is broken
    - path buffer is too small


14. astar_path
--------------
Signature:
    static int astar_path(int sx, int sy, int gx, int gy, Pt *out, int outmax)

Purpose:
    Finds a path between two map cells while avoiding obstacles.

Parameters:
    sx, sy - start coordinates
    gx, gy - goal coordinates
    out    - output path buffer
    outmax - path buffer capacity

Returns:
    Path length on success, -1 if no path exists.

How it works:
    This is a standard A* search over the 50x50 grid.

    Initialization:
        Every node gets large initial g and f values,
        parent coordinates set to -1,
        and open/closed flags reset.

    Start node:
        g = 0
        f = heuristic(start, goal)
        open = true

    Main loop:
        - scan the whole grid to find the open node with smallest f
        - if that node is the goal, reconstruct the path
        - otherwise mark it closed
        - relax all valid 8-direction neighbors

Heuristic:
    Uses Chebyshev distance.

Notes:
    The implementation is simple but not optimized. Instead of a priority queue,
    it scans the whole grid each iteration. That is acceptable for a 50x50 map.

Returns:

16. can_do_mine
---------------
Signature:
    static bool can_do_mine(double bat)

Purpose:
    Tests whether the rover can perform mining this tick.

Parameters:
    bat - current battery

Returns:
    true if battery is at least 2, otherwise false.


17. can_do_stand
----------------
Signature:
    static bool can_do_stand(bool day, double bat)

Purpose:
    Determines whether standing is legal this tick.

Parameters:
    day - whether it is daytime
    bat - current battery

Returns:
    true if standing is allowed.

How it works:
    - During daytime, standing is always allowed because the rover can gain
      solar charge during the same tick.
    - At night, standing is allowed if the rover has at least 1 battery,
      or if the battery is already 0.

Design interpretation:
    The code allows time to pass at night with zero battery, leaving the rover
    at zero rather than going negative.

Notes:
    The bat parameter is intentionally unused during daytime.


18. apply_move
--------------
Signature:
    static double apply_move(double bat, bool day, int v)

Purpose:
    Applies the battery update caused by movement.

Parameters:
    bat - current battery
    day - whether it is daytime
    v   - speed

Returns:
    New battery value after:
        bat - move_cost(v) + solar_charge(day)
    clamped into [0, BAT_MAX].


19. apply_mine
--------------
Signature:
    static double apply_mine(double bat, bool day)

Purpose:
    Applies the battery update caused by mining.

Parameters:
    bat - current battery
    day - whether it is daytime

Returns:
    New battery value after:
        bat - 2 + solar_charge(day)
    clamped into [0, BAT_MAX].


20. apply_stand
---------------
Signature:
    static double apply_stand(double bat, bool day)

Purpose:
    Applies the battery update caused by standing still.

Parameters:
    bat - current battery
    day - whether it is daytime

Returns:
    New battery value after the standing rule is applied.

How it works:
    Daytime:
        battery becomes bat - 1 + 10, clamped to valid range

    Nighttime:
        if battery >= 1, subtract 1
        otherwise remain at 0

Notes:
    This function encodes the special night rule where a rover with too little
    energy to pay standing cost simply stays at zero.


21. simulate_trip_plan
----------------------
Signature:
    static FeasResult simulate_trip_plan(int start_tick, double start_bat,
                                         int out_steps, bool mine_pending,
                                         int in_steps, int total_ticks_limit)

Purpose:
    Simulates whether a future trip plan is feasible before the mission ends.

Parameters:
    start_tick        - tick where simulation begins
    start_bat         - battery at simulation start
    out_steps         - remaining steps to reach target
    mine_pending      - whether mining still needs to happen
    in_steps          - remaining return steps after mining
    total_ticks_limit - mission end tick

Returns:
    A FeasResult:
        ok       - true if the plan can be completed
        end_tick - tick where completion or failure happened
        end_bat  - battery at that point

Plan format:
    The simulated plan is always:
        1. move outward
        2. optionally mine once
        3. move inward

Why it matters:
    This is one of the most important functions in the program.
    It is used to:
        - decide whether a mineral target is safe
        - decide whether mining now is safe
        - decide whether a speed choice preserves future feasibility

How it works:
    For each future tick until mission end:
        1. If the whole plan is already finished, return success.
        2. Generate candidate legal actions:
               STAND
               MOVE with speed 1, 2, or 3
               MINE
        3. Reject any candidate that would leave too little time to finish
           the remaining plan.
        4. Among remaining candidates, choose the one with the highest
           resulting battery. If tied, choose the one with more progress.
        5. Apply that chosen action.
        6. Advance tick and continue.

Important design point:
    This is a greedy feasibility simulator, not a full exhaustive search.
    It does not try every possible action sequence. Instead, it chooses the
    locally best action that still preserves eventual success.

Strengths:
    - battery-aware
    - time-aware
    - day/night-aware
    - fast enough for repeated use

Limitations:
    Because it is greedy, it may reject some plans that a full optimal search
    could possibly complete.


22. pick_target_safe
--------------------
Signature:
    static bool pick_target_safe(const Rover *r, const Mission *m,
                                 int *out_tx, int *out_ty)

Purpose:
    Chooses the best mineral target that the rover can safely reach, mine,
    and return from before mission time runs out.

Parameters:
    r       - current rover state
    m       - mission state
    out_tx  - output target x coordinate
    out_ty  - output target y coordinate

Returns:
    true if a safe target is found, false otherwise.

How it works:
    The function scans the whole map.
    For each mineral cell:
        - skip it if not actually a mineral
        - skip it if home is unreachable from that cell
        - compute exact outward path with A*
        - get return distance from home_dist
        - run simulate_trip_plan on:
              outward path
              one mining action
              return path

    If the trip is feasible, compare it to the current best safe target.

Tie-breaking rules:
    1. Prefer shorter total round-trip distance.
    2. If tied, prefer smaller distance to the target.
    3. If tied again, prefer larger ending battery after simulation.

Why this works:
    All minerals are treated as equally valuable, so the rover chooses the
    cheapest safe trip rather than preferring one mineral color over another.


23. choose_safe_move_speed
--------------------------
Signature:
    static int choose_safe_move_speed(const Rover *r, const Mission *m,
                                      const Pt *path, int steps_to_goal,
                                      bool target_is_mineral,
                                      int target_x, int target_y)

Purpose:
    Chooses the best speed for the current movement tick while preserving
    future feasibility.

Parameters:
    r                 - current rover state
    m                 - mission state
    path              - current path (not actually used in the function body)
    steps_to_goal     - remaining path length
    target_is_mineral - whether the current destination is a mineral
    target_x, target_y- destination coordinates

Returns:
    1, 2, or 3 for a safe movement speed, or 0 if no safe move exists now.

Normal strategy:
    Daytime preference:
        2, then 1, then 3

    Nighttime preference:
        1, then 2, then 3

Reasoning:
    The code is somewhat more aggressive during the day because solar charging
    is available, and more conservative at night.

Tight-time behavior:
    If remaining mission time is exactly tight, the order changes to:
        3, then 2, then 1

How it verifies safety:
    For each candidate speed:
        - reject if speed exceeds remaining path length
        - reject if battery is insufficient
        - compute battery after moving
        - simulate the rest of the plan
        - return the first speed that keeps the remainder feasible

Note:
    The path parameter is currently unused and could be removed from the
    function signature without changing behavior.


24. sim_step
------------
Signature:
    static void sim_step(Rover *r, const Mission *m, FILE *log_csv, FILE *log_ai)

Purpose:
    Executes exactly one simulation tick.

Parameters:
    r       - rover state, updated in place
    m       - mission state
    log_csv - CSV log file
    log_ai  - text log file

Behavior overview:
    In each tick, the rover does one of:
        1. DIGGING
        2. MOVING toward a safe mineral
        3. MOVING home
        4. STANDING

Detailed logic:

    Step 1: Mine current cell if safe
        If the rover is already on a mineral cell and home is reachable,
        it checks whether mining right now is legal and whether mining now
        still allows guaranteed safe return before mission end.
        If yes:
            - increase the correct mineral counter
            - remove the mineral from the map
            - update battery
            - log DIGGING
            - increment tick
            - return

    Step 2: Move toward a safe mineral target
        It calls pick_target_safe to find a reachable mineral with a safe
        round trip. If one exists:
            - compute the exact A* path
            - choose a safe speed with choose_safe_move_speed
            - move along the path by v cells
            - update rover position, distance, and battery
            - log MOVING
            - increment tick
            - return

    Step 3: Fall back to home return
        If no safe mineral move is possible now and the rover is not home:
            - compute A* path home
            - choose a safe homeward speed
            - move if possible
            - log MOVING
            - increment tick
            - return

    Step 4: Stand
        If nothing else is safe, the rover stands still:
            - update battery with apply_stand
            - log STANDING
            - increment tick

Why this function is central:
    sim_step is the main policy function. It turns the planning helpers into
    actual rover behavior at runtime.

Design philosophy:
    The rover is cautious. It only mines or moves when safety checks say the
    future plan is still feasible.


25. load_map_csv
----------------
Signature:
    static int load_map_csv(const char *fname, Mission *m)

Purpose:
    Loads the map file and initializes map arrays and start position.

Parameters:
    fname - CSV file name
    m     - mission structure, where start position will be stored

Returns:
    1 on success, 0 on failure.

Expected input format:
    A 50-line CSV file where each row contains 50 comma-separated cells.

Recognized symbols:
    S - start/home
    # - obstacle
    B - blue mineral
    Y - yellow mineral
    G - green mineral
    . - free cell

How it works:
    For each row:
        - read a line with fgets
        - split tokens with strtok using commas and line endings
        - inspect the first character of each token
        - fill mapc and mapv

Encoding behavior:
    S -> free cell in mapv and stored as mission start
    # -> obstacle
    B/Y/G -> mineral
    . -> free
    unknown -> treated as free '.' for robustness

Notes:
    The function requires that at least one S be found.
    If multiple S cells exist, the last one silently becomes the actual start.
    Unknown symbols do not cause failure, which is robust but can hide input
    mistakes.


26. main
--------
Signature:
    int main(void)

Purpose:
    Initializes the simulation, runs it until mission time expires,
    and prints the final result.

Main steps:

    1. Read mission length from standard input
       The program asks:
           "Küldetés hossza (óra, >=24): "
       If the value is below 24, it forces it up to 24.

    2. Convert hours into ticks
       total_ticks = hours * TICKS_PER_HOUR

    3. Load map
       Calls load_map_csv("mars_map_50x50.csv", &m)
       If loading fails, print an error and exit.

    4. Precompute home distances
       Calls precompute_home_dist on the start position.

    5. Initialize rover state
       - start at home
       - battery = 100
       - no collected minerals
       - tick = 0
       - moved_cells = 0

    6. Open output files
       Creates:
           rover_log.csv
           ai_route.txt
       If file opening fails, print an error and exit.

    7. Write initial state log entry
       The code logs round -1 as a pre-simulation initial snapshot.

    8. Run the simulation loop
       While current tick is less than mission total ticks:
           sim_step(...)

       Important:
           The simulation runs until time expires, not until battery reaches 0.

    9. Close files and print summary
       The program checks whether the rover finished at home and prints:
           - mission success/failure
           - collected minerals
           - final battery
           - names of output files

Return value:
    0 on successful completion, 1 on early failure.


Overall Architecture
--------------------
The code can be viewed as four cooperating subsystems:

1. Environment model
   - map representation
   - day/night timing
   - battery consumption and solar charging

2. Navigation
   - BFS for exact distance-to-home
   - A* for exact obstacle-aware pathfinding

3. Safety planning
   - future trip simulation
   - safe target selection
   - safe speed choice

4. Execution loop
   - one-tick decision making
   - logging
   - final mission summary

The most important design idea is that the rover does not simply chase the
closest mineral greedily. Instead, it repeatedly checks whether a future plan
is still safe before committing to movement or mining.


Strengths
---------
- Clear decomposition into small helper functions
- Good use of BFS and A* together
- Strong safety logic through trip-feasibility simulation
- Day/night charging is handled consistently
- Logging format is structured and easy to process
- The rover always prioritizes being able to return home


Possible Improvements
---------------------
1. A* performance
   The A* implementation scans the whole grid to find the next best open node.
   A priority queue would be more efficient for larger maps.

2. Greedy feasibility simulation
   simulate_trip_plan is practical and fast, but not exhaustive. A more
   complete planner could possibly find feasible plans that this version misses.

3. Unused parameter
   The path parameter in choose_safe_move_speed is currently unused.

4. Input validation
   load_map_csv silently treats unknown symbols as free cells, which may hide
   mistakes in the input map.

5. Multiple start cells
   If the map contains more than one 'S', the last one wins silently.
   A stricter parser could reject that case.

6. Mixed language
   Source comments and user-facing messages mix English and Hungarian.
   Standardizing one language would improve consistency.
